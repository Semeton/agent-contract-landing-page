the problem

AI coding agents don't remember. Your rules shouldn't depend on them doing so.

Today's AI coding workflows rely on conventions that agents are asked — not required — to follow. The model reads a file, interprets it as best it can, and may or may not comply. There is no mechanism that enforces anything.

AGENTS.md and CLAUDE.md are better than nothing. But they are text files the model reads if it decides to. Nothing stops an agent from writing to a directory it was told to leave alone, refactoring a module it was told not to touch, or ignoring conventions it was instructed to follow. You find out in the diff.
the solution

The repository is the source of truth. Not the conversation.

Instead of asking an AI to follow rules, Agent Contract makes those rules executable. Verification is deterministic — shell scripts that exit non-zero on violations. Not suggestions. Constraints.

how it works

From repository to governed agent in six steps.

Setup runs once per service. Governance runs automatically on every agent session thereafter.

01

Initialize

Run agent-contract init in any repository. Creates a .agent/ directory with the contract scaffold, hook scripts, and role definitions.

~2 min
02

Detect framework

The CLI reads your project structure and identifies language, framework, ORM, and test runner. Evidence-based detection across 14+ stacks — Node, PHP, Go, Python, Ruby, Elixir, and more.

automatic
03

Generate contract

.agent/contract.yaml is written with scope definitions, per-stack conventions, and verification rules. Idempotent — re-run any time your stack changes.

deterministic
04

Assign roles

Define roles — generator, reviewer, tester — each with explicit file scopes. A generator scoped to src/payments/** cannot write to auth, billing, or infrastructure. An agent cannot widen its own scope.

declarative
05

Run agent

The agent reads the contract before it reads your source. It loads the latest session handoff, the codebase map, and its role. The PreToolUse hook then arms enforcement for every subsequent write.

any agent
06

Verify changes

Every file write is checked against scope in real time. After the task completes, post-generate.sh runs linting, type-checking, and tests using the tools specific to your stack.

blocks violations
capabilities

What the contract encodes.

Each capability exists to solve a specific failure mode in AI-assisted development. None of these are optional nice-to-haves — they are the reason the tool exists.

roles

Role-based permissions

Different tasks get different access. A generator writes source; a reviewer reads everything and writes nothing. Roles are declared in the contract, not in a prompt that disappears.

scopes

File scopes

Each role declares the exact paths it can write to, using glob patterns. An agent assigned to payments cannot edit auth, billing, or infrastructure — regardless of what it is instructed to do mid-session.

detection

Framework detection

The CLI reads your project structure and identifies your stack automatically. It then configures the correct linter, type-checker, and test runner for post-task verification — no manual setup.

conventions

Convention learning

The contract captures your project's naming patterns, module boundaries, and coding standards. These become checks that run after every task — not suggestions the model may or may not follow.

memory

Repository memory

A Stop hook writes a handoff note after every session that produces changes. The next agent boots with the decisions, context, and outstanding issues from the previous session — not a blank slate.

hooks

Lifecycle hooks

pre-generate.sh validates the task spec and arms the active role. post-generate.sh runs verification. Each is a plain shell script that exits non-zero on failure — the universal interface for orchestrators.

verification

Stack-aware verification

After a task completes, the contract runs the right tools for your stack: eslint + tsc + jest for TypeScript, phpstan + pint + phpunit for Laravel. Verification is not generic — it is specific to what you're building.

map

Codebase map

The contract generates a structural map of your repository — modules, entry points, domain boundaries. The agent reads this before it reads a single source file, so it understands the architecture first.

principles

The engineering values behind the design.

Software engineering has always required governance — for humans, for systems, and for the processes that connect them. AI coding agents are systems. They should be governed like systems.

01

Discoverable

The contract lives in the repository alongside the code it governs. Any engineer, any agent, any CI system can find it by looking at the project root. No external configuration required.

02

Declarative

Rules are written once in a versioned file — not repeated in every prompt. The contract is the specification. The agent reads it; it does not negotiate it or interpret it differently each session.

03

Verifiable

Compliance is not self-reported. Every task ends with deterministic checks: scope verification, linting, type-checking, tests. Exit codes are the interface — not log messages, not model output.

04

Repository-first

The repository is the unit of trust, not the conversation. Rules that live only in prompts disappear when the session ends. Rules that live in the repository persist with the code — and are subject to the same review process.

Stop repeating instructions.
Put them where they belong.

The repository is already the source of truth for your code, your tests, and your CI configuration. Your AI coding rules should live there too — inside your repository, versioned, verifiable, and enforceable.
